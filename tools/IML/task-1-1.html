Edge Case Handling
3 Critical Edge Cases — Explicitly Mapped

Each edge case has a defined trigger condition, visual treatment, resolution path, and a clear owner. No silent failures.

📶
Edge Case 01 · High Priority
Offline Data Conflict
FO completes a review offline. By the time they sync, the customer's transaction data has been updated server-side — e.g., a large cash-in transaction occurred after the FO's review.
01
System compares offline draft timestamp vs. server data version hash at sync time.
02
If server data is newer: "DATA CONFLICT" flag is added to the application bundle in Manager's queue.
03
FO's recommendation is preserved but flagged. Manager receives a visual warning that FO reviewed stale data.
04
Manager sees a side-by-side panel: data at time of FO review vs. current server data. Must acknowledge before deciding.
05
Manager can proceed with conflict acknowledged or (rare path) request FO re-review before deciding.
⚠️
Edge Case 02 · Medium Priority
Sync Failure / Data Loss Risk
Sync fails repeatedly — network drops, server timeout, or device goes back offline. FO recommendation is stuck in local queue and may never reach the Manager queue.
01
Recommendation stored in encrypted local queue with exponential backoff retry: 1 min → 5 min → 15 min.
02
After 3 failed attempts: FO sees "Sync Failed" error state with specific error code, cause, and manual retry button.
03
After 6 hours without successful sync: System proactively alerts Manager that a pending recommendation may be delayed.
04
FO data is never deleted from device until confirmed server receipt. Max local storage: 30 applications.
05
On sync restore: queued items upload in sequence with server-side de-duplication check to prevent double submissions.
📊
Edge Case 03 · Low Priority
Partial Transaction Data (<3 months)
New customer or a data gap in the MFS ledger means only 1–2 months of transaction history is available instead of the standard 3-month window used for risk scoring.
01
System detects data completeness at fetch time. Flags the application: "1 of 3 months available."
02
Transaction chart rendered with explicit grey/hatched fill for missing months. No extrapolation or estimated fills.
03
Risk score shown with a lower confidence qualifier: "Based on limited data (1 month)." Eligibility uses conservative estimate.
04
FO note field shows optional prompt: "Limited history available — please add additional context from your field visit."
05
Manager sees "PARTIAL DATA" badge. Auto-batch-approve eligibility is restricted — partial-data cases must be reviewed individually.
KPI Design
One Primary KPI Per Persona

KPIs were selected to directly address the stated pain points, not vanity metrics. Each has a formula, baseline, and design lever.

▣ Field Officer · Primary KPI
Recommendation Alignment Rate (RAR)
The percentage of Field Officer recommendations that match the Manager's final decision (approve-match + reject-match). A high RAR means the FO is correctly reading risk signals and submitting quality recommendations — directly addressing the "Decision Noise" pain point where inconsistent recommendations lead to high rejection rates at the Manager level.
RAR = (Matched Decisions ÷ Total Submitted Recommendations) × 100
Target: ≥ 78% within 90 days of system launch.
Estimated baseline: ~51% (inferred from current high Manager rejection rate).
Design lever: Pre-computed AI summaries + simplified trend indicators replace raw 3-month data interpretation. Mandatory field notes force FOs to articulate their reasoning, which self-corrects decisions before submission.
▣ Manager · Primary KPI
Decision Throughput Rate (DTR)
The number of final decisions (approve + reject) processed per manager per day, without increasing the override rate beyond threshold. A Manager approving 200 applications/day with a 40% FO override rate is not performing well — they're compensating for poor FO recommendations rather than genuinely processing faster. DTR adjusts for quality.
DTR = Total Decisions Per Day ÷ Active Manager Count (adjusted: FO override rate ≤ 15%)
Target: +35% increase in decisions/day vs. current baseline.
Estimated baseline: ~80–100 decisions/manager/day under current fragmented process.
Design lever: Batch processing for Low-risk + FO-Approved cases. Risk-tier sorting in queue (most urgent first). Structured FO rationale reduces Manager's need to dig into every sub-menu.
UI State Definitions
Required System States

Each state has a defined trigger, visual treatment, and recovery path. No state is a dead end.

⬜
Loading / Skeleton
Triggered on data fetch. Skeleton screens (not spinners) maintain layout continuity so FOs and Managers aren't disoriented. Content loads progressively: profile first → transactions → risk score. Shimmer animation signals activity without blocking the UI.
◻
Empty State
Triggered when all applications in queue are processed or on first login. Never shows a blank screen. FO empty state: "All caught up!" — Manager empty state: "Queue cleared — great work." Both include a next-action prompt and a refresh hint to check for new submissions.
✕
Error / Sync Failed
Triggered on network failure, server timeout, or data corruption. Shows specific error code + human-readable cause. Provides: manual retry, contact support, and "data is safe" confirmation. Error persists until explicitly dismissed or resolved — never auto-dismissed. FO work is never lost.
◑
Partial Data
Only 1–2 months of transaction data available. Persistent amber banner above data section. Missing months displayed as grey/hatched fills in charts — no extrapolation. Risk score shown with confidence interval qualifier. Both FO and Manager see the identical flag. Never silently treated as complete.
Design Rationale
Journey Architecture Rationale
The journey was structured around a data fidelity principle: the system must never silently fail or make assumptions. Every edge case (offline conflict, sync failure, partial data) has an explicit visual flag, a defined resolution path, and a clear owner. This prevents the transparency gap where Managers make uninformed decisions because FO context was lost in transit.

The offline-first architecture was chosen because field connectivity cannot be guaranteed. Rather than blocking FOs when offline, the system accepts the trade-off of potential data staleness and surfaces it explicitly to the Manager — who is the final decision-maker and can absorb that complexity. This preserves FO productivity without sacrificing Manager confidence.

The swim-lane format was chosen to explicitly separate system responsibilities. A common failure in multi-actor design is ambiguity about who owns which step. The Manager being "absent" from Phases 01–04 is intentional design communication — it tells engineers and stakeholders that FO autonomy in the recommendation phase is a feature, not a gap in the workflow.

The KPIs were selected as lagging indicators of the UX interventions, not leading ones. RAR (Recommendation Alignment Rate) will only improve if FOs are making better-informed decisions — which is the direct outcome of the AI summary, progressive disclosure flow, and mandatory note mechanism. DTR (Decision Throughput Rate) will only improve if Managers can trust FO recommendations enough to batch-approve — which requires RAR to improve first. The two KPIs are causally linked.

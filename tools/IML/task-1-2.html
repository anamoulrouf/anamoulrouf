System Architecture Flow
Edge Cases & Resolution Logic
Offline Data
Full Offline Mode
TRIGGER: No network connectivity
  • App serves all cached application data from local SQLite store
  • "Saved Offline" icon displayed prominently on every screen
  • Data freshness timestamp shown ("Last synced 2h ago")
  • All officer interactions (notes, recommendation) saved locally with timestamp
  • Sync queue accumulates actions with sequence IDs
  • On connectivity restore: automatic background sync with progress indicator
Sync Conflict
Data Conflict on Sync
TRIGGER: Server data changed while officer was offline
  • Server detects timestamp conflict on receipt of offline data
  • System holds the submission — does NOT auto-resolve
  • Officer receives notification: "Application updated while offline"
  • Diff view presented: officer's local version vs. server version
  • Officer must explicitly re-confirm or modify recommendation
  • Both versions logged in audit trail with resolution note
Partial Data
Incomplete Transaction History
TRIGGER: <3 months of transaction data available
  • Yellow banner shown on Transaction screen: "1 of 3 months available"
  • Available months displayed normally; missing months shown as gray placeholders
  • Risk tier shown with asterisk (*) indicating data limitation
  • Officer's note field pre-populated with partial data flag prompt
  • Manager sees same partial data flag in review detail
  • System may reduce auto-eligible amount for partial data applicants
Stale Data
Data Freshness Warning
TRIGGER: Cached data >24 hours old
  • Orange banner: "Data last updated 26h ago. Verify before recommending."
  • Officer can force-refresh if connectivity exists
  • If offline, officer must acknowledge staleness before proceeding
  • Staleness flag included in recommendation metadata sent to Manager
  • Manager sees staleness indicator in application detail
Overdue Application
SLA Breach Risk
TRIGGER: Application pending >48 hours without decision
  • Application highlighted in red in Manager queue
  • Clock countdown shown: "Due in 2h"
  • Manager and their supervisor both notified via push
  • Application auto-elevated to top of Manager's queue
  • If breached: system logs SLA violation for performance tracking
Risk Override
High Risk + Approve Recommendation
TRIGGER: Officer recommends Approve for High Risk applicant
  • Officer note field becomes MANDATORY (cannot submit without explanation)
  • Warning modal: "This applicant is High Risk. Your note will be reviewed."
  • Manager sees ⚠️ flag: "Officer overrode risk score — see notes"
  • Manager cannot batch-approve these; must review individually
  • If Manager approves: both officer and manager IDs logged in audit
Sync Failed — Error Resolution

Sync Failure Recovery (Officer)

  • Error state shown: "Sync failed. Your data is saved locally."
  • Retry button available. Exponential backoff (1s, 2s, 4s, 8s)
  • Officer can continue working — all offline saves are durable
  • After 3 failed retries: "Contact support if issue persists" + error code
  • Local data NEVER deleted until server confirms receipt

Sync Failure Recovery (System)

  • Dead-letter queue for failed sync attempts
  • Operations team alerted if queue depth exceeds threshold
  • All local actions have unique idempotency keys — safe to retry
  • Server rejects duplicate submissions gracefully
  • Full sync log available in admin panel for audit
Primary KPI Mapping
Field Officer
Recommendation Acceptance Rate
RAR
Measures the percentage of a Field Officer's recommendations (Approve or Reject) that are agreed with by the Manager without override. A high RAR indicates the officer's judgment is well-calibrated with risk criteria — reducing decision noise and Manager cognitive load.

This KPI directly addresses the "Decision Noise" pain point. The design improves RAR by: (1) surfacing clear AI-generated transaction summaries that replace raw data interpretation, (2) providing structured note fields that force officers to articulate their reasoning, and (3) feeding Manager overrides back to officers as calibration signals.
TARGET → RAR ≥ 75% (baseline typically ~50-55% for unassisted officers)
MEASURED → Weekly rolling average per officer
Manager
Average Decision Time per Application
ADT
Measures the average time (in minutes) from when a Manager opens an application to when they submit a final decision. Lower ADT means higher throughput — critical when managers handle 100s of applications daily.

This KPI directly addresses the "Transparency Gap" pain point. The design improves ADT by: (1) surfacing Officer recommendation + structured notes on first load (no sub-menus), (2) visual risk flags that let managers instantly identify high-priority cases, (3) batch processing for clear-cut Low Risk / Approve Rec cases, and (4) skeleton UI that eliminates perceived wait time.
TARGET → ADT ≤ 3 minutes (vs. typical 8-12 min without structured recommendations)
MEASURED → Median + P90 across all decisions per day

Design Rationale

Why separate Recommend / Approve roles?

  • Field Officers have ground-truth context (customer behavior) but limited risk modeling capability
  • Managers have risk framework context but no field visibility
  • Separation creates checks-and-balances: reduces fraud risk, meets regulatory separation-of-duties requirements
  • Design honors this by: making "Recommend" visually distinct from approval language on mobile

Why mandatory notes for Risk Overrides?

  • High Risk + Approve Rec is the highest fraud/bad debt vector
  • Forced articulation makes officer's reasoning auditable
  • Manager can spot collusion patterns in override notes over time
  • Creates accountability without blocking the flow

Why local-first offline architecture?

  • Field connectivity is unreliable — any server-dependent action risks data loss
  • Local-first means officers never lose work, regardless of connectivity
  • Idempotency keys prevent duplicate submissions on retry
  • Trade-off: adds conflict detection complexity, but this is non-negotiable given field conditions

Why AI-generated transaction summaries?

  • 3 months of raw transaction data = cognitive overload for field officers
  • Officers need pattern signals, not data points: "spending stable," "cash-out spike in month 2"
  • Summaries standardize the inputs to Officer decisions → reduces recommendation variance
  • Risk: over-reliance on AI summary. Mitigation: full raw data always accessible below the fold

Why Officer override feedback?

  • When a Manager overrides an Officer recommendation, the Officer never learns why — this is the current "calibration blindspot"
  • Feeding manager's reason code back to officer creates a learning loop
  • Over time: officers with high RAR = candidates for promotion/trust expansion
  • System becomes self-improving, not just a workflow tool

Why batch processing for Managers?

  • 100+ applications/day with individual review = unsustainable
  • Low Risk + Approve Rec cases are high-confidence — batch approval is rational
  • Constraint: batch NEVER includes High Risk or conflict-flagged cases
  • Audit trail maintains individual decision records even for batch approvals

Core Design Philosophy: Trust with Verification

The entire system is designed around the principle that Field Officers are trusted professionals, not data entry clerks. The mobile experience amplifies their judgment (surfacing summaries, flags, and guided notes) rather than replacing it. The Manager experience is designed for verification speed, not detailed re-review — because a good recommendation with clear notes should take the Manager 60 seconds, not 10 minutes. The system earns trust by being transparent: every flag, every override, every note is shown, never hidden. This creates a decisioning layer that is both fast AND auditable.

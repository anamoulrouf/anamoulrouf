§ 01 — KPI MAPPING
📱
Field Officer KPI
Recommendation Accuracy Rate
% of Field Officer recommendations that align with Manager final decisions. High alignment = reduced cognitive overload and better structured data collection. Poor alignment = decision noise in the pipeline.
TARGET: ≥ 85% alignment rate · Measured: weekly cohort analysis
🖥️
Manager KPI
Application Throughput Time
Average time (minutes) from Manager opening an application to final decision (Approve/Reject). Drives operational efficiency and reduces applicant wait time. Batch processing is a direct lever.
TARGET: ≤ 4 min avg per application · Measured: daily session telemetry

NO CONFLICT — AUTO RESOLVE
Local version accepted as authoritative
Field Officer's local timestamp is newer than server record → system accepts and syncs automatically. FO sees "Sync Complete" status. Application proceeds to Manager queue without delay.
CONFLICT DETECTED — ESCALATION PATH
Conflict flagged, FO shown diff view
Server has a newer version (e.g., data updated at HQ) → FO is shown a "Data Conflict" modal with side-by-side comparison of their version vs server version. FO must explicitly confirm or re-review. Manager is notified of conflict history in the application detail.
⚠ CRITICAL EDGE CASE 01
Stale Data Warning
If cached application data is older than 4 hours, the FO sees a prominent "Stale Data" banner with the last sync timestamp. They cannot submit without acknowledging the warning.
System checks cache age on application open
If >4hrs: show amber "Data may be outdated" banner
FO taps to acknowledge before reviewing
Warning is logged and surfaced to Manager
⚠ EDGE CASE 02
Partial Transaction Data (1 of 3 months)
When only 1 or 2 months of transaction history exist, the trend cards show available data with explicit partial data warning. Risk tier is recalculated with available data and confidence score shown.
Detect incomplete data at fetch time
Show "Partial Data (1/3 months)" badge
Display confidence indicator on risk score
Manager sees same warning in detail view
⚠ CRITICAL EDGE CASE 03
Sync Failure / Server Error
If sync fails after connectivity is restored (server error, timeout), the app retries 3× with exponential backoff. After 3 failures, FO is shown an error state with manual retry option. Data is preserved locally.
Retry attempt 1: immediate
Retry attempt 2: after 30 seconds
Retry attempt 3: after 2 minutes
After 3 failures: manual retry + IT alert
ℹ EDGE CASE 04
Manager Overrides FO Recommendation
Manager approves an application the FO recommended rejecting (or vice versa). System logs the discrepancy. This feeds into FO accuracy tracking and may trigger coaching workflow.
Discrepancy automatically flagged in audit log
FO notified of override outcome
Aggregated monthly in FO accuracy report
Patterns trigger automated coaching prompt
⚠ EDGE CASE 05
Application Timeout / Abandoned
If a Field Officer opens an application but doesn't submit within 48 hours, it auto-returns to unassigned queue with an "Abandoned" tag. The application can be reassigned to a different FO.
24hr: FO receives reminder notification
48hr: Application auto-returns to queue
Manager notified of abandonment
Reassigned with priority escalation
ℹ EDGE CASE 06
Batch Rejection (Manager)
Manager selects multiple low-tier applications for batch rejection. System requires a batch rejection reason (dropdown + optional notes). Each application gets individual audit entries. Notifications sent in bulk.
Manager selects 2–50 applications
Confirms batch action with reason code
System creates individual audit entries per app
Bulk customer notifications triggered
